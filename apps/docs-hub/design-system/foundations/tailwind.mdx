## Overview

The `@nebutra/custom-ui` package ships a Tailwind preset that makes design tokens available as Tailwind utility classes. All color, spacing, border-radius, shadow, font, and animation values in the preset are derived from the same token source as the React components — keeping the two systems in sync.

Source: `packages/custom-ui/src/tailwind.preset.ts`

```ts
import nebutraPreset from "@nebutra/custom-ui/tailwind.preset";
```

The preset exports:

| Export | Type | Description |
|---|---|---|
| `nebutraColors` | Object | Primary, secondary, neutral + semantic colors |
| `nebutraSpacing` | Object | Named spacing scale (xs → 3xl) |
| `nebutraBorderRadius` | Object | Radius scale (none → full) |
| `nebutraShadows` | Object | Elevation shadows (sm → xl) |
| `nebutraTypography` | Object | Font families and size scale |
| `nebutraAnimations` | Object | Named CSS animation shorthands |
| `nebutraPreset` | Object | Full combined preset (default export) |

## Design Rationale

Tailwind CSS 4 supports CSS-native configuration. Rather than maintaining a separate `tailwind.config.ts` per app, the preset centralizes all token decisions. Applications consume the preset and gain access to all tokens without re-declaring them.

The preset co-locates closely with the design system tokens (`default.ts`, `brand.ts`) but is not a direct re-export — it uses a Tailwind-idiomatic structure optimized for class generation, not theme context.

<Info>
For Tailwind CSS 4, import styles directly via CSS `@source` rather than requiring a JS config file. See the installation section below.
</Info>

## Usage Guidelines

### Tailwind CSS 4 (CSS-first)

```css
/* app/globals.css */
@import "tailwindcss";
@source "@nebutra/custom-ui/styles/globals.css";
```

### Tailwind CSS 3 (Config-based)

```ts
// tailwind.config.ts
import { nebutraPreset } from "@nebutra/custom-ui/tailwind.preset";
import type { Config } from "tailwindcss";

export default {
  presets: [nebutraPreset],
  content: ["./src/**/*.{ts,tsx}"],
} satisfies Config;
```

### Programmatic access

When you need token values in TypeScript (not as Tailwind classes):

```ts
import { nebutraColors, nebutraSpacing } from "@nebutra/custom-ui/tailwind.preset";

const brandBlue = nebutraColors.primary[500]; // "#3B82F6"
const sectionPadding = nebutraSpacing["2xl"];  // "3rem"
```

## Do / Don't

**Do** use the preset instead of redefining design tokens in per-app Tailwind config.

**Do** reference semantic color tokens (`primary`, `secondary`, `success`, `error`) in utility classes rather than raw hex-based shade names.

**Don't** add one-off color values to an app's Tailwind config. Add them to `nebutraColors` in the preset and document the addition.

**Don't** mix Tailwind utility classes with inline `style={{ color: "#..." }}` values for the same property — this creates specificity conflicts and undermines the token system.

**Don't** import `nebutraPreset` in production application code paths. It is a build-time configuration artifact, not a runtime dependency.
