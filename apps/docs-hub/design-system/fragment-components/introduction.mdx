Fragment components are pre-assembled combinations of atom components for specific business contexts.
Unlike UI Patterns (structural guidelines), Fragments are **concrete, importable components**.

<Info>
  Fragment components are implemented in `@nebutra/custom-ui` and `@nebutra/design-system`.
  Import paths are specified on each component page.
</Info>

## What makes a Fragment?

A Fragment is distinguished from a primitive atom by its **composition**. Each Fragment:

- Combines two or more atoms (Button, Input, Badge, Icon, etc.) into a cohesive unit
- Encodes domain logic — it knows what it represents (an empty state, a metric card, a confirmation dialog)
- Ships with sensible defaults so it works out of the box with minimal props
- Is accessible by default (ARIA roles, keyboard navigation, focus management)

## Design principles

**Composition over configuration.** Fragments compose atoms rather than reimplementing them.
If you need a button inside a fragment, it renders the `Button` atom — not a styled `<button>`.

**Escape hatches via `className`.** Every Fragment accepts a `className` prop for one-off overrides
without forking the component.

**Controlled and uncontrolled modes.** Where state is involved (modals, multi-selects), Fragments
support both controlled (`value` / `onChange`) and uncontrolled (internal state) modes.
