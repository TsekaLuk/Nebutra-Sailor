## Overview

The color system uses a three-layer architecture that separates concerns across scale, semantics, and context.

| Layer | Source | Purpose |
|---|---|---|
| **Primitives** | `default.ts → colorPrimitives` | Raw palette — gray, blue, green, yellow, red, purple (each 0–9) |
| **Semantic** | `default.ts → semanticColorsLight / semanticColorsDark` | Role-based tokens used by all UI components |
| **Marketing** | `marketing.ts → marketingGradients / marketingEffects` | High-impact visual tokens for landing pages only |

Brand overrides live in `brand.ts → createTheme()`. The `BrandOverrides` interface accepts partial semantic color maps, so only the tokens that change need to be specified.

## Design Rationale

Semantic tokens decouple visual decisions from component code. A component that references `canvas.default` automatically responds to dark mode — no conditional logic required. Primitives are never referenced directly in components; they exist solely to feed the semantic layer.

The marketing layer is isolated by design. Gradient and glassmorphism tokens are unsuitable for application UI and should never leak into shared components.

## Marketing Color Tokens

<Warning>
Marketing tokens are for landing page surfaces only. Never import `marketingGradients` or `marketingEffects` into shared component code.
</Warning>

## Creating a Brand Theme

```ts
import { createTheme } from "@nebutra/design-system/theme/brand";

const brandTheme = createTheme("light", {
  colors: {
    accent: {
      fg: "#your-brand-color",
      emphasis: "#your-brand-emphasis",
    },
  },
  typography: {
    fontFamily: {
      normal: '"YourBrandFont", Inter, sans-serif',
    },
  },
});
```

The `BrandOverrides` interface accepts:
- `colors` — partial `semanticColorsLight` map
- `typography.fontFamily` — `normal`, `mono`, `display`
- `radii` — partial radii map
- `shadows` — arbitrary shadow record

## Usage Guidelines

Always consume colors through semantic tokens, never raw hex values or primitives.

```tsx
// Correct — semantic token via CSS variable
<div style={{ color: "var(--fgColor-default)" }} />

// Correct — Tailwind utility mapped to semantic token
<p className="text-foreground" />

// Wrong — hardcoded hex bypasses theming
<div style={{ color: "#1f2328" }} />
```

## Do / Don't

**Do** reference `canvas`, `fg`, `border`, `accent`, `success`, `attention`, `danger`, and `neutral` tokens for all UI surfaces.

**Do** isolate marketing gradient and glow effects to `apps/landing-page` components only.

**Do** use `createTheme()` to apply brand overrides rather than overwriting CSS variables manually.

**Don't** import `colorPrimitives` values (`gray.5`, `blue.4`) in component files — use semantic aliases.

**Don't** add new one-off color values to component styles. Add them to the semantic layer first.

**Don't** use marketing gradients on interactive controls (buttons, inputs, selects).
