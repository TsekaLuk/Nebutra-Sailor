## Overview

Copywriting guidelines govern every word users read: button labels, error messages, empty states, tooltips, onboarding prompts, and marketing headlines. Consistent UI copy reduces cognitive load and builds trust.

Two distinct surfaces have different writing registers:

| Surface | Register | Priority |
|---|---|---|
| Product UI | Concise, functional, calm | Clarity and task completion |
| Marketing | Compelling, specific, confident | Conversion and credibility |

These guidelines apply to both. The principles are shared; the tone adjusts to context.

## Design Rationale

Users do not read UI — they scan. Every additional word is a word that competes for attention. Good UI copy is invisible: it confirms what the user already expects and stays out of the way when things go well.

When things go wrong, copy becomes the most important design element on the screen. A clear error message recovers the user. An unclear one loses them.

## Voice

The Nebutra voice is:

- **Direct** — Say what you mean. Avoid hedging (`try to`, `you might want to`, `in some cases`).
- **Specific** — Replace vague qualifiers with concrete information.
- **Human** — Write for a person, not a compliance checklist. Avoid jargon unless the user is technical and the term is precise.
- **Confident without arrogance** — Do not over-explain or apologize unnecessarily.

What the voice is not:
- Casual to the point of being flippant
- Overly formal or legalistic
- Enthusiastic in ways that feel hollow (`Awesome!`, `Amazing!`, `Great job!`)

## Tone by Context

### Errors and failure states

Tone: neutral, direct, recovery-focused.

State what happened, why it happened (if knowable), and what to do next. Never blame the user.

| Wrong | Correct |
|---|---|
| "Something went wrong." | "Your payment could not be processed. Check your card details and try again." |
| "Invalid input." | "Email address must include @." |
| "Error 422." | "We could not save your changes. Your session may have expired — sign in again." |

### Empty states

Tone: helpful, action-oriented.

Empty states are an opportunity to guide the user toward their first action. Do not describe the absence — suggest what to do.

| Wrong | Correct |
|---|---|
| "No projects found." | "Create your first project to get started." |
| "Nothing here yet." | "Invite teammates to collaborate on this workspace." |

### Confirmations and success

Tone: brief, affirming, then out of the way.

```
File deleted.       ← Correct (past tense, 2 words)
Your file has been successfully deleted!  ← Wrong (verbose, hollow)
```

### Destructive actions

Tone: calm, precise, irreversibility is factual — not alarming.

Make the consequence clear without creating panic.

```
Delete project?
This cannot be undone. All tasks, files, and history will be permanently removed.

[Cancel]  [Delete project]
```

### Buttons and CTAs

- Use verb-noun pairs: **Create project**, **Delete account**, **Invite member**
- Match the button label to the dialog title or action being confirmed
- Avoid generic labels: **OK**, **Submit**, **Confirm** — these force the user to re-read context

| Wrong | Correct |
|---|---|
| Submit | Save changes |
| OK | Got it |
| Confirm | Delete project |
| Yes | Remove member |

### Loading and progress

- Use present progressive: **Uploading…**, **Saving…**, **Processing payment…**
- If the wait is long, give a specific reason: **Analyzing 3,400 records…**
- Do not use **Loading…** as a default — be specific about what is loading

## Usage Guidelines

### Capitalization

- **Sentence case** for all UI copy: headings, button labels, tooltips, messages
- **Title Case** for product names, feature names, and proper nouns only
- **ALL CAPS** only for overline labels and badges — never for body text

```
Wrong: Save Changes   ← Title case on button
Correct: Save changes

Wrong: this is a section heading
Correct: This is a section heading
```

### Punctuation

- No period at the end of button labels, headings, or tooltips
- Use periods for error messages and instructional body text
- Use em dashes (—) for parenthetical statements, not double hyphens (--)
- Avoid ellipsis in body copy; use it only for truncated text or in-progress states

### Numbers and units

- Spell out numbers one through nine; use numerals for 10 and above
- Use abbreviated units: 2 MB, 14 days, 3 users — no periods in abbreviations
- Use commas in large numbers: 1,200 records

### Time and dates

- Use relative time for recent events: 3 minutes ago, Yesterday
- Use absolute dates for events older than 7 days: Feb 14, 2026
- Use 24-hour time for technical contexts; 12-hour with AM/PM for general audiences

## Marketing Copy

Marketing copy follows the same voice but with a higher energy ceiling.

**Headlines:**
- Lead with the outcome, not the feature
- Use specific numbers when available: "Deploy in 30 seconds", "99.99% uptime"
- Avoid superlatives without evidence: "the best", "world-class", "unmatched"

**Subheads and body:**
- Support the headline with a specific, believable claim
- One idea per paragraph
- End with a clear CTA

**CTAs:**
- First-person framing converts better: **Start my free trial** vs **Start free trial**
- Specificity reduces friction: **See pricing** vs **Learn more**

## Do / Don't

**Do** use sentence case for all UI copy, headings, and button labels.

**Do** write error messages that state the cause and the recovery action.

**Do** match button labels to the exact action being performed — especially in destructive confirmation dialogs.

**Do** use specific, concrete language in marketing copy. Back claims with numbers.

**Don't** use filler phrases: "Please note that…", "It looks like…", "We noticed that…"

**Don't** apologize excessively. One genuine apology in an error message is appropriate. `We're sorry. We're sorry this happened. We apologize for the inconvenience.` is not.

**Don't** use passive voice for errors or instructions. "Your payment was declined" → "We could not process your payment."

**Don't** use exclamation points in error states or warnings. Reserve them for genuine celebrations (successful onboarding, first milestone achieved).
